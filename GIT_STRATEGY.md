Выбранная стратегия: GitHub Flow
Обоснование выбора:
Характеристики проекта:

Размер команды: 3 человека
Частота релизов: средняя (1-2 раза в месяц + hotfix при необходимости)
Критичность: средняя (не финтех, не медицина, но репутация важна)
Тип продукта: веб-сервис с ML-моделью
Почему GitHub Flow:

✅ Простота — команда небольшая, не нужна сложность Git Flow с множеством долгоживущих веток

✅ Быстрая итерация — DS-команда часто экспериментирует, нужна гибкость для быстрого мерджа улучшений

✅ Continuous Deployment — каждый мердж в main = потенциальный релиз, что идеально для ML-проектов

✅ Code Review — обязательные Pull Requests обеспечивают качество (важно для ML-кода)

✅ Простой hotfix — создал ветку от main, пофиксил, смержил → готово

❌ Почему НЕ Git Flow: слишком сложно для маленькой команды (ветки develop, release, hotfix избыточны)

❌ Почему НЕ Trunk-based: для ML-проектов нужны эксперименты в отдельных ветках, коммиты напрямую в main рискованны
